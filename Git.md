**Git** - распределенная система управления версиями (VCS).

Распределенная система - каждый разработчик имеет полную копию репозитория (истории проекта) на своем компьютере.

**Staging area (она же index)** — это промежуточный слой между рабочей директорией и коммитом. Есть три состояния файлов:
	1. **Working directory (рабочая директория)**  файлы, которые ты редактируешь
	2. **Staging area (index)**  
    → изменения, которые попадут в следующий commit
	3. **Repository (история / коммиты)**  
    → уже сохранённые снимки
Git добавил её, чтобы:
1. Делать **атомарные коммиты** (логически завершённые)
2. Разделять изменения
3. Управлять историей более точно

**Коммит (commit) в Git** — это зафиксированное состояние проекта в определённый момент времени. Проще говоря, это “снимок” (snapshot) файлов, которые ты решил сохранить в истории.

Git:
1. Берёт изменения из **staging area** (то, что добавлено через `git add`)
2. Сохраняет их как новый снимок проекта
3. Добавляет запись в историю

Каждый коммит содержит:
- **Изменения файлов** (не буквально diff, а снимок состояния)
- **Хэш (SHA-1)** — уникальный идентификатор
- **Сообщение (commit message)** — описание изменений
- **Автор и время**
- **Ссылку на родительский коммит**

Ветка в Git — это простой перемещаемый указатель на один из коммитов.
HEAD - это указатель на текущую локальную ветку.

Если выполняются изменения в одном и том же месте одного и того же файла в разных ветвях одного репозитория, то при попытке объединить эти ветки можно получить **конфликт слияния**.

**Rebase**

Есть основная ветка `master` и есть ветка `experiment`, в которой делаем правки.
Перебазирование этой ветки относительно ветки `master` следующим образом:

```console
$ git checkout experiment
$ git rebase master
First, rewinding head to replay your work on top of it...
Applying: added staged command

$ git checkout master
$ git merge experiment
```

`rebase` берёт все коммиты из одной ветки и в том же порядке применить их к другой ветке. При этом изменяются хэши коммитов.

**Частые вопросы:**

Отличие `git fetch` от `git pull`

git fetch и git pull служат для получения изменений из удаленного репозитория, но делают это по-разному. Fetch только **скачивает** новые данные, не меняя рабочую копию, что безопасно. Pull **скачивает и сразу сливает** (merge) эти изменения в текущую ветку. 

**Основные отличия:**

- **Git Fetch:** Скачивает коммиты, файлы и ссылки из удаленного репозитория, обновляя состояние `origin/master`, но не затрагивает ваши локальные файлы. Позволяет увидеть, что сделали другие, перед тем как объединить изменения.
- **Git Pull:** Это, по сути, комбинация `git fetch` + `git merge`. Она автоматически обновляет локальную ветку.

Отличие `merge` от `rebase`

`merge` сохраняет историю и создаёт merge-коммит, а `rebase` переписывает историю, перенося коммиты поверх другой ветки.  
В команде обычно используют `merge`, а `rebase` — для приведения ветки к чистому виду перед слиянием.

`Никогда не делай rebase публичной ветки`

Нельзя делать rebase публичной ветки, потому что он переписывает историю и меняет хэши коммитов.  
Если ветка уже запушена и используется другими разработчиками, это приведёт к рассинхронизации и проблемам при pull.
