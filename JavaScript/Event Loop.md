Event Loop (Событийный цикл) - это механизм JavaScript-окружения, который позволяет однопоточному JavaScript выполнять асинхронный код, не блокируя основной поток.

Во внутренней реализации данного механизма можно выделить следующие основные части: call stack, web api и очередь задач (task queue), которая в свою очередь делится на macrotask queue и microstask queue.

К микротаскам относятся - промисы, queueMicrotask и mutationObserver
К макротаскам относятся - Таймеры (setInterval, setTimeout), события.

После выполнения синхронного кода event loop **сначала выполняет все microtasks**, затем браузер может сделать перерисовку, и только потом берётся следующая macrotask.

Пример:

```javascript
console.log('start');

setTimeout(() => console.log('timeout'), 0);

Promise.resolve().then(() => console.log('promise'));

console.log('end');
```

Результат:

```txt
start
end
promise
timeout
```

Благодаря этому JavaScript остаётся отзывчивым, а UI не блокируется.

---

Данная архитектурная необходимость базируется на двух главных целях: сохранить консистентность (актуальность) данных и минимизировать нагрузку на графический движок браузера.

## 1. Почему микрозадачи (Microtasks) идут сразу после синхронного кода?

Микрозадачи (в основном это `Promise.then/catch/finally`, `async/await` и `queueMicrotask`) созданы для асинхронного управления данными.

Представь, что твой синхронный код запустил цепочку промисов, которая должна рассчитать финальное состояние приложения (например, отфильтровать массив, обновить стейт и посчитать итоговую сумму).

- Если бы Event Loop сразу после синхронного кода пошел делать перерисовку (Render), пользователь увидел бы на экране промежуточное, «половинчатое» состояние интерфейса, так как данные из промисов еще не успели примениться.
- Если бы Event Loop вместо микрозадач взял следующую макрозадачу (например, обработал клик пользователя), этот клик опирался бы на устаревшие данные.

Вывод: Микрозадачи выполняются до упора (пока очередь не опустеет) для того, чтобы приложение пришло в абсолютно консистентное и финальное состояние по данным до того, как эти данные кто-то увидит или начнет с ними взаимодействовать.

---

## 2. Почему перерисовка (Render) идет ПОСЛЕ микрозадач, но ДО макрозадач?

Браузерная перерисовка — это очень «дорогая» операция для процессора и видеокарты (она включает этапы Style, Layout, Paint и Composite). Чтобы сайт не тормозил, браузер стремится делать перерисовку с частотой экрана (обычно 60 Гц, то есть раз в 16.6 миллисекунд).

Event Loop устроен так, чтобы накапливать изменения.

- Синхронный код изменил цвет банда на красный.
- Микрозадача тут же изменила цвет банда на синий.
- Event Loop доходит до стадии Render. Браузер видит финальный синий цвет и рисует его один раз. Пользователь даже не узнает, что банд на наносекунду становился красным.

Если бы макрозадачи (например, `setTimeout`) выполнялись без паузы на рендер, они могли бы бесконечно забивать поток, и браузер просто завис бы, не имея возможности обновить картинку (экран бы «окуклился»).

_Важное уточнение: Браузер делает перерисовку не в каждом витке Event Loop, а только тогда, когда пришло время обновить кадр (примерно каждые 16мс) и если на странице вообще что-то изменилось._

---

## 3. Почему следующая макрозадача (Macrotask) ждет в самом конце?

Макрозадачи (события ввода, таймеры, сетевые ответы) — это глобальные, изолированные друг от друга внешние события.

Они специально изолированы шагом рендера, чтобы интерфейс оставался отзывчивым. Между выполнением двух разных макрозадач (например, обработкой движения мыши и таймером) браузер обязан иметь возможность перерисовать экран, иначе пользователь столкнется с жесткими фризами интерфейса.

---

## Короткое резюме:

1. Синхронный код + Микрозадачи — подготавливают и финализируют данные (чтобы всё было честно и без багов).
2. Рендер — один раз красиво отображает эти данные на экране (оптимизация ресурсов).
3. Макрозадача — берется за следующее независимое внешнее событие.

## Кейс 1: Бесконечный цикл в микрозадачах (Вкладка умирает 💀)

Попробуем запустить бесконечную цепочку промисов. Мы создаем микрозадачу, которая внутри себя порождает новую микрозадачу, и так до бесконечности.

```javascript
function infiniteMicro() {
  Promise.resolve().then(() => {
    infiniteMicro(); // Рекурсивно добавляем новую микрозадачу
  });
}
infiniteMicro();
```

Что происходит в Event Loop:

1. Выполняется синхронный вызов `infiniteMicro()`. В очередь микрозадач падает первый `.then()`.
2. Синхронный код завершен. Event Loop переходит к шагу «Очистить очередь микрозадач».
3. Он берет первую микрозадачу, выполняет её, и она тут же бросает в _эту же самую очередь_ следующую микрозадачу.
4. Правило Event Loop гласит: «Не уходить с шага микрозадач, пока очередь полностью не станет пустой».

Результат: Event Loop застревает на шаге микрозадач навсегда. Он никогда не дойдет до фазы Render (интерфейс замерзает, анимации останавливаются, на клики сайт не реагирует) и никогда не возьмет следующую макрозадачу. Вкладка браузера намертво вешается, и её можно только закрыть через диспетчер задач.

## Кейс 2: Бесконечный цикл в макрозадачах (Вкладка живет живчик 🏃‍♂️)

Теперь сделаем то же самое, но через макрозадачи, используя нулевой таймер `setTimeout`.

```javascript
function infiniteMacro() {
  setTimeout(() => {
    infiniteMacro(); // Планируем новую макрозадачу
  }, 0);
}
infiniteMacro();
```

Что происходит в Event Loop:

1. Выполняется синхронный вызов `infiniteMacro()`. Браузер регистрирует таймер.
2. Когда таймер готов, в очередь макрозадач встает колбэк.
3. Event Loop берет эту макрозадачу из очереди и выполняет её.
4. В процессе выполнения функция `setTimeout` планирует _новую_ макрозадачу. Но она падает в конец очереди.
5. Макрозадача завершена. И вот тут срабатывает ключевое правило: за один виток Event Loop выполняет только ОДНУ макрозадачу из очереди.
6. После этого Event Loop со спокойной совестью идет дальше: проверяет микрозадачи (они пустые), заходит в фазу Render (если нужно обновить кадр), дает пользователю кликнуть по кнопке, и только на _следующем витке_ берет из очереди ту самую новую макрозадачу, которую мы запланировали.

Результат: Процессор будет загружен, но страница останется абсолютно живой. Текст будет выделяться, анимации будут крутиться, кнопки будут нажиматься. Браузер успевает перерисовывать экран между каждым шагом нашей «бесконечной» рекурсии.

## А что будет с `requestAnimationFrame`?

Если запустить бесконечный цикл через rAF (`function loop() { requestAnimationFrame(loop); } loop();`), страница тоже будет идеально живой, так как новые вызовы rAF строго блокируются до следующего обновления экрана (каждые 16 мс), оставляя кучу времени для выполнения макрозадач и обработки действий пользователя.
