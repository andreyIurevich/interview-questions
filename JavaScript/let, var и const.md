**Карманная шпаргалка: `var` против `let` / `const`**

Главное отличие между ними заключается в области видимости, механизме всплытия (hoisting) и поведении на этапе создания переменных в памяти.

---

## 🗺️ 1. Область видимости (Scope)

- `var` — Функциональная (или глобальная) область видимости.  
    `var` игнорирует любые блоки кода, ограниченные фигурными скобками `{...}` (такие как `if`, `for`, `while`). Переменная, объявленная через `var` внутри цикла или условия, будет видна снаружи этого блока кода.
    
    ```javascript
    if (true) {
      var x = 10;
    }
    console.log(x); // 10 (Переменная «вырвалась» наружу)
    ```
    
- `let` / `const` — Блочная область видимости.  
    Эти переменные живут строго внутри тех фигурных скобок `{...}`, в которых они были объявлены. За пределами блока они физически не существуют.
    
    ```javascript
    if (true) {
      let y = 10;
    }
    console.log(y); // ReferenceError: y is not defined
    ```
    

---

## 🚀 2. Всплытие (Hoisting) и Временная мертвая зона (TDZ)

Это самый популярный глубокий вопрос на интервью. На самом деле всплывают все переменные, но ведут себя совершенно по-разному.

- Поведение `var`:  
    Движок JS на этапе компиляции находит `var`, поднимает её объявление наверх функции и автоматически выделяет под неё память, инициализируя значением `undefined`. Поэтому к `var` можно обратиться _до_ строки её объявления в коде.
    
    ```javascript
    console.log(a); // undefined (Магия всплытия)
    var a = 5;
    ```
    
- Поведение `let` / `const`:  
    Они тоже всплывают (движок знает о них заранее), но они не инициализируются значением `undefined`. Память под них зарезервирована, но читать её запрещено до тех пор, пока движок не дойдет до строки объявления в коде.  
    Период от начала блока кода до строки, где переменная объявлена, называется Временной мертвой зоной (Temporal Dead Zone — TDZ). Если обратиться к переменной в этой зоне — будет жесткая ошибка.
    
    ```javascript
    // --- НАЧАЛО TDZ ---
    console.log(b); // ReferenceError: Cannot access 'b' before initialization
    // --- КОНЕЦ TDZ ---
    let b = 5; 
    ```
    

---

## 👯 3. Повторное объявление (Redeclaration)

- `var` позволяет дублировать переменные. Ты можешь объявить переменную с одним и тем же именем сколько угодно раз. Движок просто проигнорирует повторные `var` и будет перезаписывать значения. Это часто приводило к случайным багам в больших файлах.
    
    ```javascript
    var user = "Иван";
    var user = "Петр"; // Никаких ошибок, старый user затерт
    ```
    
- `let` / `const` строго это запрещают. Попытка создать переменную с уже существующим именем в одной и той же области видимости вызовет синтаксическую ошибку на этапе парсинга кода.
    
    ```javascript
    let admin = "Иван";
    let admin = "Петр"; // SyntaxError: Identifier 'admin' has already been declared
    ```
    

---

## 🌐 4. Привязка к глобальному объекту (`window`)

- Если объявить `var` на самом верхнем уровне скрипта (вне функций), она автоматически становится свойством глобального объекта (в браузере — `window.имя_переменной`). Это загрязняет глобальную область видимости.
    
    ```javascript
    var globalVar = "Я везде";
    console.log(window.globalVar); // "Я везде"
    ```
    
- Переменные `let` и `const`, объявленные на верхнем уровне, не создают свойств в объекте `window`. Они сохраняются во внутреннем скрытом контексте декларативных областей видимости.
    
    ```javascript
    let globalLet = "Я скрыт";
    console.log(window.globalLet); // undefined
    ```
    

---

## 🔄 5. Поведение внутри циклов (Повторение Задачи №10)

- `for (var i = 0; ...)` — создает одну общую переменную `i` на весь цикл в памяти. Все асинхронные колбэки (таймеры, события), созданные внутри цикла, будут смотреть в эту единственную ячейку памяти и увидят её финальное состояние по окончании цикла.
- `for (let i = 0; ...)` — на каждой итерации цикла движок JavaScript создает абсолютно новое лексическое окружение со своей независимой копией переменной `i`. Каждая функция замыкает свою собственную переменную.

---

## 🦾 6. Кратко: В чем разница между `let` и `const`?

Они работают абсолютно одинаково с точки зрения блочной видимости и TDZ, но отличаются правилами перезаписи:

- `let` — значение переменной можно изменять сколько угодно раз.
- `const` — константа. Ей необходимо присвоить значение сразу при объявлении, и её ссылку в памяти нельзя изменить в будущем.
- Ловушка: `const` защищает от перезаписи только саму ссылку. Если в `const` записан объект или массив, его внутренние свойства/элементы можно изменять (иммутабельность является поверхностной).
    
    ```javascript
    const arr = [];
    arr.push(1); // Отлично работает, ссылка на массив не изменилась!
    arr =;   // TypeError: Assignment to constant variable (Попытка сменить ссылку)
    ```

## 🎯 Резюме для ответа на собеседовании:

> «Я всегда использую `const` по умолчанию для неизменяемых ссылок, и `let`, если переменная должна менять значение (например, в циклах). Использование `var` в современном коде является устаревшей практикой (anti-pattern), так как функциональная видимость, отсутствие защиты от повторного объявления и скрытое поведение всплытия (hoisting) часто становятся источниками трудноотлавливаемых багов».

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log('Вывод:', i);
  }, 1000);
}
```

## Академический разбор ловушки с `var`

## 1. Порядок выводов ➡️ `3, 3, 3`

Ты абсолютно прав. Из-за правил Event Loop, которые мы разбирали в Задаче №5, все три макрозадачи `setTimeout` встают в очередь и ждут полной очистки Call Stack. Цикл завершается намного быстрее, чем проходит 1 секунда. На момент, когда цикл полностью отработал, переменная `i` успела дорасти до `3`.

## 2. В чем разница между `var` и `let` (для интервью):

- `var` имеет функциональную (или глобальную) область видимости. У `var` нет блочной области видимости (он игнорирует фигурные скобки `if` или `for`). Поэтому на весь цикл создается ровно одна переменная `i` в памяти. Все три стрелочные функции внутри `setTimeout` замыкаются на одну и ту же ячейку памяти. Когда таймеры срабатывают, они смотрят в эту общую ячейку и видят там финальную тройку.
- `let` имеет блочную область видимости. Ты сформулировал это отлично: для каждой итерации цикла создается своя независимая копия переменной `i`. Под капотом движок JavaScript на каждом шаге цикла создает новое лексическое окружение и копирует туда текущее значение. Каждый таймер замыкает свою собственную, персональную переменную `i`, которая больше никогда не изменится.

## 3. Два способа исправления:

Способ А (Современный через `let`):

```javascript
for (let i = 0; i < 3; i++) { // Просто меняем var на let
  setTimeout(() => console.log('Вывод:', i), 1000);
}
```

Способ Б (Исторический через IIFE — Immediately Invoked Function Expression):  
До появления ES6 и `let` разработчики были вынуждены вручную создавать новую область видимости для каждой итерации, оборачивая код в немедленно вызываемую функцию:

```javascript
for (var i = 0; i < 3; i++) {
  (function(localI) { // Переменная localI становится изолированной в рамках этой функции
    setTimeout(() => {
      console.log('Вывод:', localI);
    }, 1000);
  })(i); // Передаем текущее значение i как аргумент
}
```
