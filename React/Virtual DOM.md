**Virtual DOM (VDOM)** — это **внутреннее представление UI в памяти** (обычные JS-объекты), которое описывает, **как должен выглядеть DOM**, а не сам DOM.

Каждый рендер компонента создаёт **новое виртуальное дерево**. Затем React:

1. Сравнивает новое дерево со старым (**reconciliation / diffing**)
2. Вычисляет минимальный набор изменений
3. Применяет эти изменения к реальному DOM (**commit phase**)

**Почему операции с реальным DOM считаются «дорогими»?**

Сами по себе JavaScript-объекты работают молниеносно, но взаимодействие с браузерным DOM ресурсоёмко по нескольким причинам:

- **Пересчет геометрии (Reflow/Layout)**: Когда вы меняете размер, положение или содержимое элемента, браузеру приходится заново вычислять координаты и размеры **всех** затронутых элементов на странице. Если изменить один элемент в начале страницы, это может вызвать цепную реакцию пересчета для всего документа.
- **Перерисовка (Repaint)**: После пересчета геометрии браузер должен заново отрисовать пиксели на экране (цвета, тени, границы).
- **Сложная структура**: Реальный DOM — это огромный объект с сотнями свойств и методов. Проход по такому дереву и его модификация требуют гораздо больше памяти и процессорного времени, чем работа с плоскими JS-объектами в Virtual DOM.

**Итог:** Virtual DOM не делает работу с реальным DOM быстрее, он делает её **реже**. Вместо того чтобы перестраивать всё здание при замене одного окна, React находит это окно на чертеже и меняет только его.
