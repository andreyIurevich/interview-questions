Двойной вызов эффектов в `StrictMode` — это, пожалуй, главный кошмар каждого разработчика, который впервые запускает React в режиме разработки 😅 Кажется, что в коде баг, но на самом деле это фича, созданная для спасения твоего приложения.

Давай разберем, зачем React это делает и какие проблемы он пытается подсветить.

---

## Главная причина: Поиск сайд-эффектов и утечек памяти

Начиная с React 18, в режиме разработки (`development`) внутри компонента `<React.StrictMode>` React намеренно производит цепочку действий:

> Монтирование → Мгновенное размонтирование → Повторное монтирование

Он делает это с одной единственной целью: убедиться, что твой компонент умеет правильно за собой «прибираться».

В идеальном приложении на React компоненты должны быть предсказуемыми. Если компонент смонтировался, а затем размонтировался, он должен вернуть систему в исходное состояние (не оставлять запущенных таймеров, открытых WebSocket-соединений или подписок на глобальные события). Если функция очистки (`cleanup`) написана плохо или отсутствует, двойной вызов сразу же вскроет эту проблему.

---

## Примеры: что ломается, если не написать очистку?

React специально провоцирует баги в `dev`-режиме, чтобы ты заметил их до того, как код улетит на прод.

## 1. Подписка на глобальные события (Window Event Listeners)

```jsx
// ТАК ДЕЛАТЬ НЕЛЬЗЯ
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // Забыли написать return () => window.removeEventListener(...)
}, []);
```

- Без StrictMode: При старте добавится 1 слушатель. Всё работает. Но если пользователь перейдет на другую страницу и вернется, слушателей станет 2, потом 3 (утечка памяти!).
- Со StrictMode: Из-за двойного монтирования у тебя сразу же создастся 2 слушателя на старте. Ты откроешь вкладку _Performance_ или начнешь дебажить и мгновенно увидишь, что функция `handleResize` срабатывает дважды на один скролл. Баг обнаружен за 5 секунд.

## 2. Запуск интервалов (`setInterval`)

```jsx
// ТАК ДЕЛАТЬ НЕЛЬЗЯ
useEffect(() => {
  setInterval(() => console.log('Тик'), 1000);
  // Забыли clearInterval
}, []);
```

- Со StrictMode: Счетчик начнет тикать в два раза быстрее (в консоли будет вылетать по два «Тика» каждую секунду), потому что первый интервал остался жить в памяти браузера, а второй наложился сверху.

## Что еще делает StrictMode дважды?

Помимо хуков `useEffect` и `useLayoutEffect`, StrictMode в режиме разработки запускает два раза:

- Тело самого функционального компонента (функцию-рендерер).
- Функции обновления стейта (`setCount(prev => prev + 1)`).
- Аргументы-функции в хуках `useMemo` и `useCallback`.

React делает это, чтобы проверить их на чистоту (Purity). Функции рендеринга обязаны быть «чистыми» — они не должны менять глобальные переменные или внешние объекты во время вычисления Virtual DOM. Если функция чистая, то от того, что React вызовет её два раза вместо одного, интерфейс никак не изменится. Если изменился — у тебя скрытый побочный эффект в рендере.

---

## Важно помнить: На продакшене этого НЕТ

Когда ты собираешь проект для деплоя (`npm run build`), режим `StrictMode` полностью отключается. В продакшен-сборке все эффекты и рендеры будут срабатывать строго по одному разу.

Поэтому не нужно пытаться обойти это поведение костылями (например, заводить рефы вида `const isFirstRender = useRef(true)` чтобы заблокировать второй вызов). Правильный путь — всегда писать функцию возврата (`cleanup`) в своих эффектах.
