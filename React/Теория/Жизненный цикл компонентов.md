Жизненный цикл компонента состоит из трех основных этапов: **Монтирование** (Mounting), **Обновление** (Updating) и **Размонтирование** (Unmounting).

### Жизненный цикл компонентов-классов

На каждом из этапов жизненнего цикла компонентов-классов вызывается определенная функция, в которой мы можем определить какие-либо действия:

1. constructor(props): конструктор, в котором происходит начальная инициализация компонента
    
2. static getDerivedStateFromProps(props, state): вызывается непосредственно перед рендерингом компонента. Этот метод не имеет доступа к текущему объекту компонента (то есть обратиться к объкту компоненту через this) и должен возвращать объект для обновления объекта state или значение `null`, если нечего обновлять.
    
3. render(): рендеринг компонента
    
4. componentDidMount(): вызывается после рендеринга компонента. Здесь можно выполнять запросы к удаленным ресурсам
    
5. componentWillUnmount(): вызывается перед удалением компонента из DOM
    

Кроме этих основных этапов или событий жизненного цикла, также имеется еще ряд функций, которые вызываются при обновлении состояния после начального рендеринга компонента, если в компоненте происходят обновления:

1. static getDerivedStateFromProps(props, state)
    
2. shouldComponentUpdate(nextProps, nextState): вызывается каждый раз при обновлении объекта props или state. В качестве параметра передаются новый объект props и state. Эта функция должна возвращать true (надо делать обновление) или false (игнорировать обновление). По умолчанию возвращается true. Но если функция будет возвращать false, то тем самым мы отключим обновление компонента, а последующие функции не будут срабатывать.
    
3. render(): рендеринг компонента (если shouldComponentUpdate возвращает true)
    
4. getSnapshotBeforeUpdate(prevProps, prevState): вызывается непосредственно перед обновлением компонента. Он позволяет компоненту получить информацию из DOM перед возможным обновлением. Возвращает в качестве значения какой-то отдельный аспект, который передается в качестве третьего параметра в метод `componentDidUpdate()` и может учитываться в componentDidUpdate при обновлении. Если нечего возвращать, то возвращается значение `null`
    
5. componentDidUpdate(prevProps, prevState, snapshot): вызывается сразу после обновления компонента (если shouldComponentUpdate возвращает true). В качестве параметров передаются старые значения объектов props и state. Третий параметр - значение, которое возвращает метод `getSnapshotBeforeUpdate`

**Аналоги в функциональных компонентах**

Функции-хуки позволяют повторить поведение жизненного цикла:

| `componentDidMount`     | `useEffect(() => {}, [])`                     |     |
| ----------------------- | --------------------------------------------- | --- |
| `componentDidUpdate`    | `useEffect(() => {...}, [dep])`               |     |
| `componentWillUnmount`  | `useEffect(() => { return () => {...} }, [])` |     |
| `shouldComponentUpdate` | `React.memo()` + `useCallback`, `useMemo`     |     |

**Описание этапов жизненного цикла в функциональных компонентах**

В React жизненный цикл компонентов зависит от того, как они написаны. В современной разработке функциональные компоненты с хуками полностью вытеснили классовые.

Жизненный цикл компонента состоит из трех основных этапов: Монтирование (Mounting), Обновление (Updating) и Размонтирование (Unmounting).

Ниже приведена подробная шпаргалка, как эти этапы устроены в функциональных компонентах и как они соотносятся со старыми классовыми методами.

**Этап 1: Монтирование (Mounting)**

Это этап, когда компонент рождается, инициализируется и впервые вставляется в реальный DOM.

1. Инициализация и первый рендер:
    
    - React вызывает тело твоей функции. Инициализируются стейты (`useState`) и рефы (`useRef`).
    - Строится виртуальный DOM (Virtual DOM).
    
2. Мутация DOM:
    
    - React синхронно переносит созданную структуру в реальный DOM браузера. На этом шаге экран пользователя еще не обновлен.
    
3. Вызов `useLayoutEffect`:
    
    - Синхронно выполняются все `useLayoutEffect`. Поток заблокирован, это последний шанс замерить размеры элементов до отрисовки. _(В классах этому соответствовал метод `componentDidMount`)_.
    
4. Отрисовка (Paint):
    
    - Браузер отображает интерфейс на экране.
    
5. Вызов `useEffect`:
    
    - Асинхронно (отложенно) срабатывают `useEffect` с пустым массивом зависимостей `[]`. Компонент полностью готов к работе. _(В классах это тоже была зона ответственности `componentDidMount`)_.
    

**Этап 2: Обновление (Updating)**

Этот этап происходит каждый раз, когда у компонента меняются пропсы (props), внутренний стейт (state) или перерендеривается его родитель.

1. Повторный рендер:
    
    - React заново выполняет тело функции-компонента с новыми данными.
    - Вычисляется новый Virtual DOM и сравнивается со старым (процесс Reconciliation).
    
2. Сброс старых эффектов (Cleanup):
    
    - Перед тем как применить новые эффекты, React должен «прибраться» за старыми.
    - Сначала синхронно вызываются функции очистки (`return () => {}`) из предыдущих `useLayoutEffect`.
    - Затем (после Paint) вызываются функции очистки из предыдущих `useEffect`.
    
3. Обновление реального DOM:
    
    - Изменения накатываются на браузерный DOM.
    
4. Вызов новых эффектов:
    
    - Синхронно отрабатывают новые `useLayoutEffect` _(в классах: `componentDidUpdate`)_.
    - Происходит Paint (отрисовка).
    - Отложенно отрабатывают новые `useEffect`, если изменились переменные в их массиве зависимостей.
    

**Этап 3: Размонтирование (Unmounting)**

Этот этап наступает, когда компонент удаляется со страницы (например, переключили вкладку или скрыли элемент по условию).

На этом этапе компонент больше не делает никаких рендеров. Происходит только одна важная вещь: тотальная очистка.

- React вызывает функции очистки (`return () => {}`) абсолютно всех запущенных `useEffect` и `useLayoutEffect`.
- Здесь нужно отписываться от глобальных событий (`window.addEventListener`), закрывать WebSocket-соединения и удалять таймеры (`clearTimeout`), чтобы не плодить утечки памяти.
- _(В классовых компонентах для этого существовал единственный метод `componentWillUnmount`)_.

## 📊 Сводная шпаргалка по хуку `useEffect`

В функциональных компонентах именно хук `useEffect` управляет поведением на разных этапах жизненного цикла в зависимости от массива зависимостей:

```jsx
useEffect(() => {
  // 1. Код тут срабатывает при МОНТИРОВАНИИ 
  // 2. И при каждом ОБНОВЛЕНИИ, если изменилось someValue

  return () => {
    // 1. Код тут срабатывает ПЕРЕД каждым новым обновлением someValue
    // 2. И окончательно при РАЗМОНТИРОВАНИИ компонента
  };
}, [someValue]); // Массив зависимостей
```

- `[]` (пустой массив): Эффект сработает 1 раз при монтировании, очистка — 1 раз при размонтировании.
- `[a, b]` (с переменными): Эффект и очистка будут перезапускаться при каждом изменении `a` или `b`.
- Без массива (вообще нет `[]`): Эффект будет жестко срабатывать при абсолютно каждом рендере компонента. (Использовать осторожно!).

**Отличия функционального подхода от классового**

1. В классе - наличия конструктора, метода render и биндинга this, в функции этого нет
2. В классе есть методы жизненного цикла - `componentDidMount`, `componentDidUpdate`, `componentWillUnmount`, в функции - один `useEffect` заменяет несколько lifecycle-методов
3. Отличие работы с состоянием (в классе - this.setState, в функции - set...)
4. Повторное использование логики (в классе - HOC, Render Props, в функции - кастомные хуки)
5. Функции короче, проще и читаемее
6. Будущее react - использование хуков
