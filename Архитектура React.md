**Основные понятия**

1. Согласование (Reconciliation)

Это **весь процесс** обновления интерфейса в React. Представьте его как общую стратегию.  
Когда состояние компонента меняется, React должен решить, как синхронизировать Virtual DOM с реальным DOM. Согласование включает в себя:

- Вызов метода `render` (создание нового дерева).
- Сравнение деревьев.
- Принятие решения: обновить старый узел, удалить его или создать новый.
- Само применение изменений в браузере.

2. Диффинг (Diffing)

Это конкретный **алгоритм сравнения**, который запускается внутри процесса согласования. Это «глаза» процесса.  
Диффинг сравнивает два дерева Virtual DOM (старое и новое) и ищет разницу. Чтобы делать это быстро, React использует две хитрости (эвристики):

1. **Тип элемента**: Если типы элементов разные (был `<div>`, стал `<span>`), React не ищет сходства, а просто сносит старую ветку и строит новую.
2. **Ключи (keys)**: При работе со списками React использует ключи, чтобы понять, какой элемент просто переместился, а какой действительно изменился.

Причиной использования эвристического алгоритма сравнения - является кубическая сложность классических алгоритмов сравнения деревьев. При данном подходе - сложность линейная.

3. Рендер - это процесс, при котором React вызывает ваши компоненты, чтобы выяснить, как должен выглядеть интерфейс в данный момент времени. С целью построения Virtual DOM.

4. **Virtual DOM (VDOM)** — это **внутреннее представление UI в памяти** (обычные JS-объекты), которое описывает, **как должен выглядеть DOM**, а не сам DOM.

**Описание архитектуры**

**React Fiber** — это современная архитектура React (появилась в v16), которая полностью пересмотрела то, как работает процесс **Reconciliation** (Согласование).

Если раньше React обновлял интерфейс «одним махом» и мог «заморозить» страницу, то Fiber сделал этот процесс гибким и прерываемым.

Главная идея: Инкрементальный рендеринг

Fiber разбивает работу по согласованию на маленькие порции (единицы работы), которые называются **файберами**. Это позволяет React:
- **Приостанавливать** рендеринг, если браузеру нужно выполнить более важную задачу (например, обработать клик или анимацию).
- **Приоритизировать** обновления. Ввод текста в input имеет более высокий приоритет, чем фоновая загрузка данных.
- **Возвращаться** к отложенным задачам позже.

Две фазы Reconciliation в Fiber

Весь процесс согласования теперь четко разделен на две части:

1. **Фаза Рендеринга (Render Phase) — Асинхронная**:
    - React обходит дерево компонентов и сравнивает новые элементы со старыми.
    - Он составляет «список изменений» (Effect List), но **ничего не меняет в браузере**.
    - Эту фазу можно прервать или перезапустить, если пришло более срочное обновление.
2. **Фаза Фиксации (Commit Phase) — Синхронная**:
	- React берет список изменений из первой фазы и разом применяет их к реальному DOM.
	- Эту фазу **нельзя прерывать**, чтобы пользователь не увидел наполовину обновленный интерфейс.

До Fiber использовался «Stack Reconciler», который работал рекурсивно. Если ваше дерево компонентов было огромным, рекурсия занимала основной поток браузера слишком долго, и анимации начинали лагать. Fiber превратил рекурсию в **связанный список**, по которому можно «гулять» и делать паузы.

Fiber — основа **Concurrent Mode**, который предоставляет более гибкие механизмы управления рендером. В этом режиме React может частично показывать обновлённый интерфейс, не блокируя остальную часть приложения.

### Преимущества Concurrent Mode

- Плавные анимации и переходы.
- Асинхронная обработка сложных вычислений без «подвисаний» интерфейса.
- Приоритетный рендер важных задач.

Хук **`useTransition`** — это прямой интерфейс к возможностям Concurrent Mode. Он позволяет пометить тяжелое обновление интерфейса как **неприоритетное** (переходное).

`useTransition` позволяет разделить:

1. **Срочное обновление:** ввод текста (должен быть мгновенным).
2. **Несрочное обновление:** фильтрация списка (может чуть подождать).

Как он используется?

Хук возвращает массив из двух элементов:

1. **`isPending`** (boolean): сигнализирует, что «фоновая» работа еще идет.
2. **`startTransition`** (function): обертка, в которую нужно поместить функцию изменения стейта.

```jsx
import { useState, useTransition } from 'react';

function SearchList() {
  const [isPending, startTransition] = useTransition();
  const [filterTerm, setFilterTerm] = useState('');

  const handleChange = (e) => {
    // Срочное обновление: текст в инпуте меняется сразу
    const value = e.target.value;
    
    // Несрочное обновление: оборачиваем в startTransition
    startTransition(() => {
      setFilterTerm(value); // Это обновление React может поставить на паузу
    });
  };

  return (
    <div>
      <input type="text" onChange={handleChange} />
      
      {isPending && <p>Загрузка списка...</p>}
      
      <HugeList term={filterTerm} />
    </div>
  );
}
```

критический нюанс:

1. Как это работает на самом деле

React ожидает, что код внутри `startTransition` выполнится **мгновенно**.

- **Правильно:** `startTransition(() => { setList(data); })` — здесь вы просто говорите React: «Запланируй обновление этого стейта с низким приоритетом».
- **Бесполезно:** `startTransition(() => { complexSorting(bigArray); })` — если вы просто запустите сортировку внутри, она все равно заблокирует основной поток, потому что это обычный JavaScript-код.

2. Где происходит «тяжелая работа»?

Тяжелая работа должна происходить **в процессе рендеринга** компонента, который потребляет этот стейт.

Когда вы делаете `setFilterTerm(value)` внутри перехода:

1. React помечает обновление как «несрочное».
2. Он начинает рендерить компоненты, которые зависят от `filterTerm`.
3. Если внутри этих компонентов происходит тяжелая сортировка (например, прямо в теле функции или через `useMemo`), React понимает, что это часть «несрочного» рендера.
4. **Магия Fiber:** Поскольку это «несрочный» рендер, React может прервать его выполнение на середине, если пользователь снова что-то ввел, и вернуться к нему позже.

Хук `useDeferredValue` - так же как `useTransition`  помечает обновление как низкоприоритетное, но с разных сторон.

Пример использования:

```jsx
function SearchPage() {
  const [query, setQuery] = useState('');
  // Создаем "отложенную" версию поискового запроса
  const deferredQuery = useDeferredValue(query);

  return (
    <>
      {/* Инпут всегда работает быстро, так как привязан к query */}
      <input value={query} onChange={e => setQuery(e.target.value)} />
      
      {/* Список результатов будет лагать меньше, так как использует отложенное значение */}
      <div style={{ opacity: query !== deferredQuery ? 0.5 : 1 }}>
        <HugeList term={deferredQuery} />
      </div>
    </>
  );
}
```

Главные отличия от useTransition

- **useTransition** используется там, где у вас есть доступ к функции **изменения** стейта (`setSomething`). Вы сами контролируете момент старта.
- **useDeferredValue** используется там, где вы получаете готовое значение «сверху» (через **props**), и не можете обернуть его изменение в `startTransition`.

Сравнение с Debounce (отложенным вызовом)

Часто `useDeferredValue` путают с дебаунсом (setTimeout). Но есть ключевое отличие:

- **Debounce** всегда ждет фиксированное время (например, 300мс), прежде чем начать работу. Это создает искусственную задержку.
- **useDeferredValue** не ждет. Он начинает рендерить «в фоне» **сразу**, как только освободится основной поток. Если компьютер мощный, задержка будет незаметна. Если слабый — React будет подстраиваться под него.
