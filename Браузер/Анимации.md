`requestAnimationFrame` (сокращённо **rAF**) — это API браузера для **плавных, синхронизированных с отрисовкой экрана анимаций**.
Это встроенный метод браузера, который вызывает переданную в него функцию в тот момент, когда браузер готовится совершить перерисовку.

```js
requestAnimationFrame(callback);
```

1. Регистрирует `callback`
2. Браузер вызывает его **перед следующим repaint**
3. Обычно ~60 раз в секунду (зависит от монитора)

Получаем **один вызов = один кадр**

Метод `requestAnimationFrame` (или сокращенно rAF) стоит особняком: он не относится ни к микрозадачам, ни к классическим макрозадачам. У него есть своя собственная, выделенная очередь, которая намертво привязана к графическому циклу монитора.

Если говорить точнее, rAF выполняется внутри фазы Render (перерисовки экрана) — сразу перед тем, как браузер начнет физически рассчитывать геометрию элементов (Layout) и красить пиксели (Paint).

## Обновленная схема Event Loop с учетом rAF:

Браузер крутит цикл Event Loop миллионы раз в секунду. Но фаза Render активируется только тогда, когда монитор готов отобразить новый кадр (при 60 Гц — каждые 16.6 мс).

Вот как выглядит виток Event Loop, в котором браузер решил обновить экран:

```unset
1. Синхронный код (Текущая макрозадача)
2. Очередь микрозадач (Выполняется до полного опустошения)
3. Фаза RENDER (Обновление экрана):
   ├── Вызовы requestAnimationFrame ◄── [ rAF ВЫПОЛНЯЕТСЯ ТУТ ]
   ├── Расчет стилей и геометрии (Style, Layout)
   └── Отрисовка пикселей (Paint, Composite)
4. Следующая макрозадача из очереди (например, setTimeout)
```

---

## 4 главных факта о работе rAF в этой цепочке:

1. Идеальный тайминг для анимаций: Так как rAF срабатывает прямо перед отрисовкой, любые изменения DOM (например, сдвиг элемента на 5 пикселей), сделанные внутри rAF, попадут на экран без задержек. Если сделать то же самое через `setTimeout`, он может выполниться неудачно — например, сразу _после_ рендера, и изменения «зависнут» в памяти еще на 16 мс, вызвав визуальный микро-фриз (jank).
2. Поведение микрозадач внутри rAF: Если внутри функции `requestAnimationFrame` ты создашь промис (микрозадачу), Event Loop не пойдет сразу к Layout/Paint. Он сначала очистит микрозадачи, созданные в rAF, и только потом продолжит рендер.
3. Защита от лишней работы: Если вкладка браузера свернута или скрыта, браузер полностью отключает фазу Render для экономии батареи и процессора. В этот момент очередь rAF замораживается и не выполняется вообще, в то время как обычные макрозадачи (`setTimeout`) продолжают крутиться (хоть и с искусственным замедлением).
4. Опустошение очереди rAF: В отличие от микрозадач, которые выполняются «до победного» (даже если они бесконечно создают новые микрозадачи), очередь rAF имеет лимит. При старте фазы Render браузер берет _только те_ колбэки rAF, которые уже были запланированы. Если внутри rAF вызвать другой rAF (`requestAnimationFrame(loop)`), он уйдет в очередь следующего кадра и не заблокирует текущий рендер.

#### Монитор

Частота вызова равна частоте обновления экрана, это обычно **60 FPS**, но может быть 90 / 120 / 144

**FPS (Frames Per Second)** — это **количество кадров в секунду**, то есть **сколько раз интерфейс перерисовывается за 1 секунду**. 

Чем выше FPS, тем **плавнее выглядит анимация**.

Это **физическое свойство монитора**, браузер **не может его изменить**.

Монитор:
1. просто «считывает» готовые кадры из видеобуфера
2. делает это строго по своему таймеру

Управление кадрами в системе:

**Браузер**
- **не рисует кадры сам**
- он **подготавливает кадры вовремя**
- отдаёт их GPU

**GPU + ОС**
- знают частоту монитора
- синхронизируют вывод кадров (VSync / compositor ОС)

#### Время на кадр

Формула расчёта времени на один кадр: frameTime (ms) = 1000 / FPS

##### Как это использовать на практике

Пример: 60 FPS
Бюджет кадра: **16.7 ms**

В это время должно уместиться:
1. JS
2. Style
3. Layout
4. Paint
5. Composite

Если любой этап «съел» больше — **кадр пропущен**.

Итого:
✔ Монитор обновляется с фиксированной частотой  
✔ Браузер подстраивает цикл рендера под эту частоту  
✔ `requestAnimationFrame` — точка синхронизации  
✔ Если кадр не готов вовремя — он пропускается  
✔ Подвисания = пропущенные кадры
