Поток выполнения js-кода в браузере основан на событийном цикле.

Событийный цикл - это бесконечный цикл, в котором движок JavaScript ожидает задачи, исполняет их и снова ожидает появления новых.

Задачи поступают - движок выполняет их, затем ожидает новые задачи. Если задача поступает, в то время когда, движок занят выполнением другой задачи, то она добавляется в очередь макрозадач (macrotask queue). Задачи из очереди исполняются по правилу «первым пришёл – первым ушёл»

Алгоритм работы:
1. Выбрать и исполнить старейшую задачу из очереди макрозадач (например, «script»).
2. Исполнить все микрозадачи:
	- Пока очередь микрозадач не пуста: Выбрать из очереди и исполнить старейшую микрозадачу
3. Отрисовать изменения страницы, если они есть.
4. Если очередь макрозадач пуста – подождать, пока появится макрозадача.
5. Перейти к шагу 1.

Алгоритм решения задач на событийный цикл:
1. Выполняется основной поток кода (+ выполняются функции в теле создания промисов)
2. Выполняются микрозадачи (обработчики в then/catch/finally)
	2.1  Если есть несколько промисов и у них несколько then/catch, то сначала группируются и выполняются все первые then/catch, затем группируются и выполняются все вторые then/catch и т.д.
	2.2  Если во время выполнения одной из микрозадач была найдена макрозадача, то она будет помещена в конец очереди макрозадач.
	 2.3  Макрозадачи выполняются по одной. Перед тем как выполнить каждую следующую задачу Event Loop проверяет нет ли активных задач из Call Stack, а также нет ли микрозадач.
3. Выполняются макрозадачи (setTimeout, mousemove и т.д.)

##### Задачи

Если во время выполнения одной из микрозадач была найдена макрозадача, то она будет помещена в конец очереди макрозадач

```javascript
console.log(1);

setTimeout(() => console.log(2));

Promise.resolve().then(() => console.log(3));

Promise.resolve().then(() => setTimeout(() => console.log(4)));

Promise.resolve().then(() => console.log(5));

setTimeout(() => console.log(6));

console.log(7);

/// Вывод: 1 7 3 5 2 6 4
```

Макрозадача, порождает микрозадачу.  
Макрозадачи срабатывают не все, а лишь по одной, а потом цикл проверяется очередь основного потока и микрозадач. Получается, что макрозадача порождает микрозадачу, выполняет ее, и лишь потом берет следующую макрозадачу.

```javascript
new Promise(resolve => setTimeout(resolve)).then(() => console.log(4));

Promise.resolve(5).then(console.log);

console.log(6);

setTimeout(() => console.log(7),0);

// Вывод: 1 6 3 5 2 4 7 
```

Каждый `.then()` создает отдельную Promise reaction. Следующий обработчик в цепочке не попадает в microtask queue до тех пор, пока предыдущий Promise не будет выполнен. Поэтому Promise-цепочки выполняются последовательно, а не все сразу.

Если есть несколько промисов и у них несколько then/catch, то сначала группируются и выполняются все первые then/catch, затем группируются и выполняются все вторые then/catch и т.д.

```javascript
console.log('1');

setTimeout(() => { console.log('2'); });

new Promise((resolve) => { resolve(); })
	.then(() => { console.log('3'); })
	.then(() => { console.log('4'); });

new Promise((resolve) => { resolve(); })
	.then(() => { console.log('5'); })
	.then(() => { console.log('6'); });

setTimeout(() => { console.log('7'); });

// Вывод: 1 3 5 4 6 2 7
```

```js

const a = 1; // Инициализация 'a' происходит после ошибки в промисе.

	.catch(() => console.log(b--)) // Выполняется, т.к. промис rejected. Выводит текущее 'b' (2), затем декрементирует 'b' до 1.
	.catch(() => console.log(0)); // Пропускается, т.к. ошибка обработана первым catch.

/**

Порядок вывода:

1) undefined ← синхронный console.log(c), c = undefined из-за hoisting

2) 2 ← микротаск .catch срабатывает, выводит b (2), затем b становится 1

3) 1 1 3 ← макротаск setTimeout: a=1, b=1 (после декремента), c=3

*/
/*
Функция в `setTimeout` не делает «снимок» (snapshot) значений в момент своего объявления. Благодаря **замыканию**, она сохраняет доступ к живым переменным и считывает их значения в самый последний момент — когда до неё доходит очередь выполнения.
*/
```

```js
async function f() { 
	console.log(1); 
	const promise = new Promise((resolve) => { 
		console.log(2); 
		setTimeout(() => { 
			console.log(3); 
			resolve('готово!'); 
			console.log(4); 
		}); 
	}); 
	console.log(5); 
	const result = await promise; 
	console.log(6); 
	console.log(result); 
	return 'Result'; 
} 

f(); 
console.log(7);

// 1 2 5 7 3 4 6 готово!
```

Это одно из самых частых заблуждений при работе с `async/await`. Кажется, что вызов `f()` должен заблокировать весь код ниже, пока функция не завершится, но на самом деле асинхронная функция не блокирует основной поток выполнения.

Ключевой секрет здесь в том, когда именно функция `f()` прерывается и возвращает управление внешнему коду.

Давай разберем по шагам, как движется Event Loop по твоему коду:

## Шаг 1: Синхронный старт функции `f()`

Когда ты вызываешь `f()`, движок заходит внутрь неё и начинает выполнять код абсолютно синхронно сверху вниз:

1. Выполняется `console.log(1)` ➔ Вывод: 1.
2. Создается `new Promise`. Как мы уже разбирали раньше, его конструктор тоже выполняется синхронно.
3. Внутри конструктора выполняется `console.log(2)` ➔ Вывод: 2.
4. Регистрируется `setTimeout`. Его колбэк улетает в очередь макрозадач (он выполнится позже).
5. Конструктор промиса завершен. Промис сейчас находится в состоянии pending (ожидание).
6. Выполняется `console.log(5)` ➔ Вывод: 5.

## Шаг 2: Встреча с `await` — точка разделения

Дальше движок доходит до строчки:

```javascript
const result = await promise;
```

Вот здесь и происходит магия. Слово `await` говорит движку: _«Так, этот промис еще не готов. Мне нужно подождать. Я временно ставлю выполнение функции `f()` на паузу, выхожу из неё и иду выполнять оставшийся внешний синхронный код»_.

Функция `f()` как бы «засыпает» на этой строчке, но основной поток кода продолжает бежать дальше.

## Шаг 3: Возврат во внешний код

Движок вышел из «уснувшей» функции `f()` и вернулся на то место, откуда её вызвали. Следующей строчкой там идет:

```javascript
console.log(7); // ➔ Вывод: 7
```

Весь синхронный скрипт на этом завершен. В консоли на данный момент мы видим: `1 2 5 7`.

## Шаг 4: Очередь макрозадач (таймер)

Синхронный код закончился, микрозадачи пусты. Event Loop заглядывает в очередь макрозадач и находит там колбэк от `setTimeout`. Он начинает выполняться:

1. Выполняется `console.log(3)` ➔ Вывод: 3.
2. Вызывается `resolve('готово!')`. Наш промис внутри `f()` мгновенно переходит в состояние fulfilled (успешно выполнен). Колбэк для продолжения функции `f()` отправляется в очередь микрозадач.
3. Код внутри таймера продолжает идти дальше (вызов `resolve` не останавливает функцию!). Выполняется `console.log(4)` ➔ Вывод: 4.

## Шаг 5: Очередь микрозадач (пробуждение функции)

Макрозадача таймера завершена. Event Loop перед следующим шагом обязан очистить микрозадачи. Там как раз лежит команда «разбудить функцию `f()`, так как её промис зарезолвился».

1. Функция `f()` просыпается на строчке с `await`. Переменная `result` получает значение `'готово!'`.
2. Выполняется `console.log(6)` ➔ Вывод: 6.
3. Выполняется `console.log(result)` ➔ Вывод: готово!.
4. Функция возвращает `'Result'`.

---

## Главный вывод

Вызов `f()` запускает функцию, но как только она доходит до первого `await` невыполненного промиса, она отдает управление обратно внешнему коду. Именно поэтому `7` выводится раньше, чем завершается внутренность `f()`.

Если ты хочешь, чтобы `7` гарантированно вывелась в самом конце, нам нужно применить `await` или `.then()` к самому вызову `f()`.

**Уточнение насчёт работы `resolve('готово!')`**

Cам вызов `resolve('готово!')` — это обычная, синхронная команда. Это не микрозадача.

Когда ты вызываешь `resolve('готово!')`, внутри движка JavaScript синхронно происходит две вещи:

1. Промис меняет свой статус с «ожидания» (pending) на «выполнен» (fulfilled).
2. Промис «смотрит» в свой внутренний список: _«Так, а кто меня ждал через `.then()` или через `await`?»_.

Он видит, что на строчке `const result = await promise;` его послушно ждет заблокированная функция `f()`.

И вот именно в этот момент сам промис берет оставшийся «кусок» функции `f()` (всё, что идет _после_ строчки с `await`) и упаковывает его в микрозадачу. Он как бы говорит Эвент Лупу: _«Я освободился! Положи, пожалуйста, в очередь микрозадач команду разбудить функцию `f()` и передай ей значение 'готово!'»_.

## Посмотрим на код глазами движка:

Чтобы было проще понять, движок под капотом виртуально переписывает твой `await` в обычный `.then()`. Вот как для него выглядит функция `f()` с момента встречи с `await`:

```javascript
// То, что было после await, превращается в колбэк для .then()
promise.then((resolvedValue) => {
  const result = resolvedValue;
  console.log(6);
  console.log(result);
  return 'Result';
});
```

## Итог для Шага 4 пункта 2:

- Вызов `resolve()` выполняется синхронно внутри таймера.
- Но результатом этого вызова становится то, что продолжение функции `f()` (код с 6-кой и выводом результата) встает в очередь микрозадач и ждет, пока завершится текущая макрозадача (таймер).

```js
console.log("1");

setTimeout(() => {
  console.log("2");
}, 0);

async function foo() {
  console.log("3");

  await null;

  console.log("4");
}

foo();

Promise.resolve().then(() => {
  console.log("5");
});

console.log("6");

// 1 3 6 4 5 2
```

`await` всегда приводит ожидаемое значение к Promise (концептуально через `Promise.resolve`). Поэтому даже `await null` приостанавливает выполнение async-функции, а её продолжение выполняется как microtask.

---

```js
Promise.resolve(1)
  .then(x => {
    console.log("A");
    return Promise.resolve(x + 1);
  })
  .then(x => {
    console.log("B", x);
  });

console.log("C");
// С, А, B 2
```

второй `.then()` ждет завершения **Promise, который вернул callback1**.

callback определяет судьбу следующего Promise в цепочке

То есть callback не просто что-то возвращает — он решает, **в каком состоянии окажется Promise, который вернул `.then()`**:

- вернул значение → следующий Promise становится `fulfilled(value)`;
- выбросил ошибку → следующий Promise становится `rejected(error)`;
- вернул Promise → следующий Promise **принимает его состояние** (fulfilled или rejected после завершения этого Promise).
